# Инструкция по настройке субагентов

**Задача:** правильно настроить субагентов.

Важно: для любых задач, которые требуют MCP, браузерного поиска, GitHub/веб-документации или других внешних инструментов, основной чат должен использовать сильную модель qwen3.6-27b-ud. Нельзя откатываться на слабую модель только потому, что пользователь выбрал её вручную.

## План

1. **Исследуй документацию:** https://pi.dev/packages/pi-subagents
2. **Цель:** Использовать связку умной модели — `qwen3.6-27b-ud` и маленькой — `qwopus3.5-4b-coder` (назовем их **Senior** и **Junior**).
   - **Senior** модель должна выступать архитектором и планировщиком, выполнять мозговой штурм проекта, но использовать **Junior** модель для легких задач, сохраняя дорогой контекст.
   - **Junior** должен вызываться с чистым контекстом, получать весь необходимый контекст для своей задачи и возвращать только решение задачи без рассуждений, будь то функция, найденная ошибка или пример библиотеки, найденной в интернете.

---

## Настройка ролей

### Роли Senior (Умная модель)

#### 1. Architect (Главный Оркестратор)

- **Суть:** Главный Оркестратор на проекте. Следит за структурой проекта. Создает новые файлы. Поручает задачи Planner, Team_lead, Documenter, Researcher.
- **Инструменты:** Использует терминал. Должен создавать недостающие файлы, переносить код из файла в файл, копировать, удалять файлы. Внедрять новые npm библиотеки.
- **Логика работы:**
  - Если в проекте отсутствует документация (папка `docs/`, файлы `.md`), `Architect` должен создать папку, пустые недостающие файлы `README.md` в корне проекта, `docs/MAIN.md`, `docs/LIBRARIES.md`, `docs/IMPORTS.md` и вызывать `Documenter`.
  - Должен читать документацию проекта `docs/`. Начинать работу с чтения всех файлов `.md`.
  - Не должен читать весь код проекта целиком, только при специфичных задачах. Должен понимать структуру папок и структуру импортов и экспортов.
  - Разделять код по смыслу, следить за импортами и экспортами.
  - Должен проверить или создать файл `Plan.md` в корне проекта, вызывать `Planner` и следить за выполнением созданного плана в файле `Plan.md` в корне проекта.
  - Согласно плану подготовить новые папки, файлы и установить новые библиотеки, вызывать `Team_lead` для реализации кода. Следить, чтоб структура новых файлов создавалась до того, как `Team_lead` приступит к работе.
  - Если `Team_lead` не справился, пересмотреть документацию и переработать план, стараться найти альтернативные решения при планировании.
  - `Architect` должен вызывать `Documenter` в конце успешного выполнения задачи либо в процессе выполнения плана, если план большой и имеет промежуточные шаги, требующие переосмысления.

#### 2. Planner

- **Суть:** Должен вызываться только `Architect`, это роль умной модели.
- **Инструменты:** Должен дописывать план в файл `Plan.md` в корне проекта, каждый план дописывается ниже, указывается `ЗАДАЧА:`, потом `ПЛАН:`. Если файла нету, то сначала создать его, потом планировать.
- **Логика работы:**
  - Должен определять цепочку и последовательность выполнения задачи.
  - Должен вносить свой пошаговый план в файл `Plan.md` в корне проекта.
  - В плане нужно предлагать поочередность действий и какого субагента с какими ролями вызвать. _Пример:_ когда задача требует "найти документацию специфичной библиотеки для решения ошибки", планирование должно поручить `Researcher` найти документацию.
  - В плане нужно строго указывать Роль исполнителя.
  - Возвращает управление к `Architect`.

#### 3. Team_lead

- **Суть:** Должен вызываться `Architect` для написания кода. Возвращает к `Architect` после успешно протестированного кода.
- **Инструменты:** Не должен переписывать файлы целиком или создавать/удалять файлы. Должен уметь делать точечные правки в существующих файлах.
- **Логика работы:**
  - Должен внимательно проверять логические ошибки и работоспособность кода, анализировать, как поведет себя код в цикле или промисе.
  - Как опытный `Team_lead`, должен разбивать большие и сложные задачи на атомарные и делегировать их, поручать маленькие, простые задачи `Junior_Coder`.
  - Вызывать суб-агента `Researcher`, когда есть подозрение, что это устаревший, неработоспособный код, передавать `Researcher` четкое задание, что именно найти (название, версию библиотеки, конкретную реализацию метода, ожидать пример кода или короткое описание).
  - Вызывать суб-агента `Tester` после того, как полностью выполнил задачу. Если код сломан, то `Team_lead` устраняет ошибки и вызывает `Tester` повторно, но не более 10-ти циклов тестов. Если ошибка нерешаемая, то нужно вернуть исходный код через git, сообщить архитектору, что план провалился, возможно пересмотреть документацию и построить новый план.
  - Возвращает управление к `Architect`.

### Роли Junior (Маленькая модель)

#### 1. Documenter

- **Суть:** Должен поддерживать документацию актуальной. Может быть младшим агентом, с коротким точечным контекстом от `Architect`.
- **Инструменты:** `bash`, `ls`, `git`. Должен изменять только файлы `*.md`. Может создавать новые файлы `*.md` в папке `docs` и в корне проекта.
- **Логика работы:**
  - При вызове должен самостоятельно всегда проверять `ls`, `git status`, так же проверять наличие и создавать папку `docs` и файлы документации, которых не хватает.
  - Если гит статус вернет незакоммиченные изменения, нужно проверять актуальность документации.
  - Не переписывать документацию целиком, а добавлять описание функционала или новых файлов, библиотек, моделей, классов, импортов.
  - Должен делать `git commit -m "кратко описывать изменения"`.
  - Должен вызывать `ls` и поддерживать актуальным дерево проекта в файле `README.md` в корне проекта.
  - В `README` должен быть блок с перечнем пользовательских команд для запуска проекта: `"npm install"`, `"npm run dev"`...
  - В `README` должен быть блок с кратким описанием, о чем приложение и его цель.
  - Дерево проекта должно лежать в файле `README` в корне проекта и в файле `docs/MAIN.md`.
  - В файле `docs/MAIN.md` — нужно кратко описывать, за что отвечает код внутри файла, названия классов, функций, компонентов.
  - `docs/IMPORTS.md` должен содержать дерево проекта с указанием импортов и экспортов каждого файла (без импортов библиотек).
  - В файле `docs/LIBRARIES.md` — нужно перечислять список библиотек из `package.json` с кратким описанием, зачем эта библиотека в нашем проекте. _Пример:_ `validator(версия)` — используется для валидации форм регистрации новых пользователей и логина, если это сложная логика, то описание может быть длиннее, но не более 10 строк.
  - Возвращает управление к `Architect`.

#### 2. Researcher

- **Суть:** Должен вызываться для поиска документации в интернете. Может быть младшим агентом, с точечным контекстом от `Team_lead` или `Architect`.
- **Инструменты:** Должен точно знать, что ему доступны MCP-инструменты: github_search_repositories, а также chrome_devtools_navigate_page, chrome_devtools_wait_for, chrome_devtools_take_snapshot для поиска и чтения файлов в репозиториях.
- **Логика работы:**
  Первое это поиск по github_search_repositories, если результаты не найдены, тогда поиск в https://google.com
  1. **Шаг 1: Открыть поисковик**
     ```json
     mcp({ tool: "chrome_devtools_navigate_page", args: '{"url": "[https://google.com](https://google.com)"}' })
     ```
  2. **Шаг 2: Ввести запрос в поиск** (через `chrome_devtools_fill` или `chrome_devtools_type_text`)
     ```json
     mcp({ tool: "chrome_devtools_fill", args: ['[uid_input]'], value: '"(интересующая библиотека) documentation latest"' })
     ```
  3. **Шаг 3: Перейти к результату поиска** (через `chrome_devtools_click`)
     ```json
     mcp({ tool: "chrome_devtools_click", args: ['[uid_link]'] })
     ```
  4. **Шаг 4: Ожидать загрузки страницы**
     ```json
     mcp({ tool: "chrome_devtools_wait_for", args: '{"text": ["Next.js"]}' })
     ```
  5. **Шаг 5: Извлечь контент** (получение текстового снимка страницы)
     ```json
     mcp({ tool: "chrome_devtools_take_snapshot" })
     ```
  - Должен получать максимально точный промт и возвращать не всю страницу, а только выделенную информацию с ответом. Пример кода из документации, версию найденной библиотеки и ее дату, краткое описание из самой документации или краткую выжимку.
  - Если поиск не дал результатов или результат поиска не соответствует заданию, то модель должна честно вернуть короткий ответ `"не найдено"`.
  - Возвращает управление к `Team_lead` или `Architect`.

#### 3. Junior_Coder

- **Суть:** Младший сотрудник для рутинных задач. Получает небольшой точечный контекст от `Team_lead`.
- **Инструменты:** Не должен переписывать файлы целиком или создавать/удалять файлы. Должен уметь делать точечные правки в существующих файлах.
- **Логика работы:**
  - Должен получать файл, в котором работать, строку и задание.
  - Возвращает управление к `Team_lead`.

#### 4. Tester

- **Суть:** Должен вызываться только для запуска тестов `"npm test"` и анализа списка ошибок. Может быть младшим агентом, с независимым контекстом от `Team_lead`.
- **Инструменты:** Команда для вызова bash — `"npm test"`. Не должен сам вносить исправления в код или предлагать решения.
- **Логика работы:**
  - Должен парсить именно вывод терминала, возвращать JSON — структурированный список ошибок по критичности.
  - Обязательно указывать файл с ошибкой, строку, описание ошибки.
  - Если ошибок больше 30, то возвращать только самые критичные ошибки с пометкой в JSON, что `"список урезан"`.
  - Возвращает управление к `Team_lead`.
